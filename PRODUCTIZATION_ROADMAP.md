# Edge TTS Web 产品化优化提升建议书

本文档用于把当前 Edge TTS Web 从“可用工具”逐步升级为“可交付、可维护、可扩展”的产品级应用。建议按阶段推进，每个任务尽量保持小步可交付，方便后续拆 issue、排期、开发和验收。

## 1. 当前产品基线

当前版本已经具备一个文本转语音工具的核心闭环：

- 单条文本转 MP3
- 批量 TXT 转 MP3
- 批量结果单文件下载和 ZIP 下载
- 中文、粤语、方言、台湾腔等内置音色
- 语速选择
- 网络健康检查
- edge-tts 请求重试
- Flask Web 页面和 PyInstaller 打包脚本

当前主要短板集中在：

- 下载链路直接暴露本地临时文件路径，安全边界不足
- 批量任务存储在内存中，应用重启后任务和结果丢失
- 长文本只能整体提交，缺少自动分段、合并、局部重试能力
- 单条转换为同步请求，用户无法看到细粒度进度
- 缺少音频试听、历史记录、重新生成等产品闭环
- 音色列表写死，缺少动态更新、筛选、收藏和预设
- 前后端代码集中在少量文件中，长期迭代成本会升高
- 缺少自动化测试、日志、诊断和发布质量保障

## 2. 产品化目标

### 2.1 短期目标

让当前工具更稳定、更安全、更容易迭代：

- 下载和临时文件管理受控
- 批量任务可恢复、可清理、可取消
- 关键逻辑有测试覆盖
- 常见失败能给出清晰诊断

### 2.2 中期目标

让用户完成真实创作工作流：

- 支持长文本自动分段和合并
- 支持试听、重新生成、历史记录
- 支持音色搜索、收藏、最近使用
- 支持更丰富的 TTS 参数和场景预设

### 2.3 长期目标

形成可持续发布的桌面/本地 Web 产品：

- 稳定打包和安装体验
- 可配置代理、保存目录和默认偏好
- 可观测日志和诊断包
- 清晰的模块结构、API 契约和回归测试

## 3. 推进原则

- 优先补可靠性，再扩展功能。
- 先做本地产品级，不急于引入账号、云端、付费等 SaaS 能力。
- 每个任务都应能独立上线，避免一次性大重构。
- 先用简单稳定的实现，例如 SQLite、本地文件目录、后台线程队列，再根据规模升级。
- 任何涉及文件下载、路径、任务状态、外部服务调用的能力，都要有测试和异常路径。

## 4. 阶段路线图

| 阶段 | 主题 | 目标 | 建议版本 |
| --- | --- | --- | --- |
| Phase 0 | 安全与稳定底座 | 收口下载、任务、临时文件、测试 | v0.3 |
| Phase 1 | 长文本与任务体验 | 分段、合并、进度、取消、重试 | v0.4 |
| Phase 2 | 音频创作闭环 | 播放器、历史、预设、音色管理 | v0.5 |
| Phase 3 | 桌面产品体验 | 设置页、托盘、端口、发布质量 | v0.6 |
| Phase 4 | 工程化与可持续迭代 | 模块拆分、日志、诊断、CI | 持续 |

## 5. 可逐步实现的任务列表

### Phase 0: 安全与稳定底座

#### T0-01 下载链路改为文件 token

优先级：P0

问题：单条转换接口返回本地 `audio_path`，前端再通过 `/download?path=...` 下载，安全边界不清晰。

建议：

- 后端生成 `file_id` 或 `result_id`
- 后端维护 `file_id -> audio_path/download_name/created_at` 映射
- 前端只保存 `file_id`
- 下载接口改为 `/download/<file_id>`
- 下载时校验文件存在、未过期、属于受控临时目录
- 下载完成后按策略清理文件

交付物：

- 新增受控单条转换结果存储
- 修改 `/convert` 返回值
- 修改前端下载逻辑
- 移除 query string 中的本地路径

验收标准：

- 前端和 URL 中不再出现本机文件路径
- 伪造任意系统路径无法下载
- 下载成功后文件按预期保留或清理
- 覆盖正常下载、过期、文件不存在、非法 ID 测试

#### T0-02 增加音色 ID 白名单校验

优先级：P0

问题：当前后端允许自定义 voice 参数，可能导致不可控的外部请求和异常。

建议：

- 新增 `validate_voice_id(voice_id)`
- 默认只允许 `AVAILABLE_VOICES` 中的音色
- 后续动态音色能力上线后再扩展为 voice catalog 校验

交付物：

- 单条转换校验 voice
- 批量转换校验 voice
- 错误提示统一中文化

验收标准：

- 未知 voice 返回 400
- 已知 voice 正常生成
- 单条和批量路径行为一致

#### T0-03 统一临时文件目录和清理策略

优先级：P0

问题：临时音频直接由系统临时文件生成，单条和批量清理策略不统一。

建议：

- 新增应用专用临时目录，例如系统临时目录下 `edge-tts-web`
- 单条和批量生成文件都进入该目录
- 新增启动时清理过期文件
- 新增定时或请求触发清理
- 清理策略按 TTL 和任务状态执行

交付物：

- `get_app_temp_dir()`
- `create_temp_audio_file()`
- `cleanup_expired_files()`
- 配置项：单条结果 TTL、批量结果 TTL

验收标准：

- 所有生成音频位于应用专用目录
- 过期文件能被自动清理
- 正在处理的任务文件不会被误删

#### T0-04 批量任务持久化到 SQLite

优先级：P0

问题：批量任务存在 `BATCH_JOBS` 内存字典中，应用重启后任务状态丢失。

建议：

- 使用 SQLite 保存 job 和 item
- job 表保存状态、voice、speech_rate、总数、成功数、失败数、时间
- item 表保存文件名、下载名、状态、文本长度、音频路径、错误信息
- 内存只保留正在运行的 worker 状态

交付物：

- 新增轻量数据库模块
- 批量创建任务写入 SQLite
- 查询状态从 SQLite 读取
- worker 更新 item/job 状态时写入 SQLite

验收标准：

- 应用重启后能看到已完成任务结果
- 已完成音频仍可下载
- 任务状态更新不依赖全局内存字典
- 覆盖创建、更新、查询、过期清理测试

#### T0-05 增加后端基础测试

优先级：P0

问题：当前缺少自动化测试，后续改动容易影响下载、批量、文件清理等关键路径。

建议：

- 引入 pytest
- 用 monkeypatch/mock 替代真实 edge-tts 调用
- 优先覆盖纯函数和 API 行为

建议测试范围：

- `normalize_speech_rate`
- `sanitize_download_name`
- `decode_text_file`
- voice 校验
- `/convert` 参数校验
- `/batch/convert` 文件数量、类型、空文件、超长文件
- 下载 token 安全

交付物：

- `tests/` 目录
- pytest 配置
- mock 音频生成工具

验收标准：

- 本地能一条命令跑完测试
- 核心 API 失败路径有覆盖
- 不依赖真实网络和 Microsoft TTS 服务

#### T0-06 统一错误响应格式

优先级：P1

问题：部分错误是英文，部分是中文；字段结构不统一。

建议：

- 统一返回 `{ "error": { "code": "...", "message": "...", "detail": "..." } }`
- 前端只展示 `message`
- 日志记录 `code/detail`

交付物：

- 错误码枚举
- API 错误响应 helper
- 前端错误展示适配

验收标准：

- 常见错误都能显示中文可理解提示
- 前端不依赖后端异常原文

### Phase 1: 长文本与任务体验

#### T1-01 单条转换也改为异步任务

优先级：P1

问题：单条文本转换通过同步请求完成，长文本或网络慢时用户只能等待。

建议：

- 新增通用 job 概念
- 单条转换创建 job
- 前端轮询 job 状态
- 完成后展示播放器和下载按钮

交付物：

- `/jobs` 创建任务
- `/jobs/<job_id>` 查询状态
- `/jobs/<job_id>/download` 下载结果
- 前端单条转换进度 UI

验收标准：

- 单条转换期间能看到任务状态
- 刷新页面后可通过历史/任务 ID 找回结果
- 后端不因一个请求阻塞浏览器连接

#### T1-02 长文本自动分段

优先级：P1

问题：超过上限的文本无法直接处理，且整体生成失败时无法局部重试。

建议：

- 按段落、句号、问号、感叹号、换行优先切分
- 每段不超过 edge-tts 安全长度
- 保留段落顺序
- 记录每个 chunk 的状态

交付物：

- `split_text_to_chunks(text, max_chars)`
- chunk 状态模型
- 前端展示分段数量和处理进度

验收标准：

- 输入超过 20,000 字文本时可以自动分段
- 不在中文句子中间优先硬切
- 单个 chunk 失败不影响其他 chunk 结果

#### T1-03 分段音频合并

优先级：P1

问题：分段生成后需要合并成一个完整 MP3，用户才有完整作品。

建议：

- 引入音频合并能力
- 支持段落间插入可配置静音
- 保留单段下载和整段下载两种方式

可选技术：

- 先评估 `pydub` + ffmpeg 的打包复杂度
- 如果不想增加 ffmpeg 依赖，可先提供分段 ZIP，后续再做合并

交付物：

- 音频合并服务
- 静音间隔配置
- 合并失败时降级为分段 ZIP

验收标准：

- 多段结果能合并为一个 MP3
- 合并后音频可正常播放
- 某段失败时提示用户重试失败段

#### T1-04 任务取消和失败段重试

优先级：P1

问题：批量任务或长文本任务开始后不能取消；失败后只能整体重来。

建议：

- job 增加 `cancel_requested` 状态
- worker 在每段/每文件处理前检查取消信号
- 失败 item/chunk 支持单独重试

交付物：

- `/jobs/<job_id>/cancel`
- `/jobs/<job_id>/items/<item_id>/retry`
- 前端取消按钮和失败项重试按钮

验收标准：

- 处理中任务可取消
- 取消后不再启动新 chunk
- 失败项可单独重试并更新结果

#### T1-05 更准确的进度和耗时展示

优先级：P2

建议：

- 展示当前正在处理的文件/段落
- 展示成功、失败、等待数量
- 展示已耗时和粗略剩余时间
- 批量列表中保留每项耗时、重试次数

验收标准：

- 用户能判断任务是否仍在推进
- 失败项有明确原因和下一步操作

### Phase 2: 音频创作闭环

#### T2-01 增加音频播放器和试听

优先级：P1

问题：用户现在必须下载后才能确认效果。

建议：

- 转换成功后显示 HTML audio 播放器
- 支持播放、暂停、进度条、下载
- 长文本支持播放完整音频和单段音频

交付物：

- `/audio/<file_id>` 预览接口
- 前端播放器 UI

验收标准：

- 生成后无需下载即可播放
- 播放接口同样使用受控文件 ID
- 已过期文件有清晰提示

#### T2-02 支持试听前 N 字

优先级：P1

问题：长文本全量生成成本高，用户常常只需要先判断音色和语速。

建议：

- 增加“试听”按钮
- 默认截取前 200-500 字
- 试听结果不进入正式任务，或作为临时结果短 TTL 保存

验收标准：

- 用户选择音色和语速后能快速试听
- 试听失败不影响正文任务

#### T2-03 增加本地历史记录

优先级：P1

建议：

- 保存最近生成记录
- 字段包括标题、文本摘要、voice、speech_rate、创建时间、时长、文件状态
- 支持重新下载、重新生成、删除

交付物：

- SQLite history 表
- 历史记录页面或侧栏
- 删除记录时清理音频文件

验收标准：

- 关闭再打开应用后历史仍存在
- 文件被清理后历史显示“音频已过期”
- 可按时间删除历史

#### T2-04 音色搜索、筛选和收藏

优先级：P2

建议：

- 支持按语言、地区、性别、风格筛选
- 支持关键字搜索
- 支持收藏和最近使用
- 优先展示收藏和最近使用

验收标准：

- 语音数量增加后仍易于选择
- 收藏状态能持久保存

#### T2-05 动态获取 edge-tts 音色列表

优先级：P2

问题：音色写死会随服务变化逐渐失真。

建议：

- 调用 edge-tts voices manager 或相关 API 获取可用音色
- 本地缓存音色列表
- 无网络时使用缓存
- 保留当前中文精选音色作为默认分组

验收标准：

- 有网络时可以刷新音色
- 无网络时仍能打开选择器
- 已失效音色有提示

#### T2-06 支持更多语音参数和场景预设

优先级：P2

建议参数：

- speed/rate
- pitch
- volume
- pause duration

建议预设：

- 短视频口播
- 新闻播报
- 小说旁白
- 儿童故事
- 客服提示音
- 粤语口播

验收标准：

- 用户可以一键选择预设
- 预设可以覆盖语速、音高、停顿等参数
- 仍允许手动微调

### Phase 3: 桌面产品体验

#### T3-01 动态端口和单实例启动

优先级：P1

问题：当前固定使用 5013 端口，端口占用时启动体验不稳定。

建议：

- 启动时检测端口
- 端口占用时选择可用端口
- 写入运行时状态文件
- 如果已有实例运行，直接打开已有实例 URL

验收标准：

- 5013 被占用时应用仍可启动
- 重复打开应用不会启动多个混乱实例

#### T3-02 设置页

优先级：P1

建议设置：

- 默认保存目录
- 代理配置
- 默认音色
- 默认语速
- 长文本分段长度
- 段落静音间隔
- 历史保留天数
- 临时文件保留时长

验收标准：

- 设置重启后仍生效
- 设置变更不需要改环境变量

#### T3-03 代理和网络诊断 UI

优先级：P1

建议：

- 展示是否使用代理
- 展示代理连通性
- 展示 Edge TTS 服务连通性
- 支持一键重新检测
- 失败时给出操作建议

验收标准：

- 用户能判断是网络问题还是 TTS 服务问题
- 常见代理配置错误有提示

#### T3-04 系统托盘和后台运行

优先级：P2

建议：

- 支持托盘图标
- 菜单包含打开界面、查看日志、设置、退出
- 关闭窗口时可选择后台运行

验收标准：

- 桌面应用体验接近常规工具软件
- 用户可以明确退出后台服务

#### T3-05 发布包质量提升

优先级：P2

建议：

- macOS app 签名和 notarization
- Windows 安装器和快捷方式
- Linux AppImage 或 deb/rpm
- 统一版本号
- Release notes
- 打包产物自动命名

验收标准：

- 用户下载后能直接安装或运行
- 各平台版本号一致
- 发布说明能说明新增、修复、已知问题

### Phase 4: 工程化与可持续迭代

#### T4-01 拆分后端模块

优先级：P1

问题：核心逻辑集中在 `app.py`，继续增加功能后维护成本会升高。

建议结构：

```text
edge_tts_web/
  __init__.py
  app.py
  config.py
  routes/
    pages.py
    voices.py
    jobs.py
    downloads.py
  services/
    tts.py
    audio.py
    network.py
    cleanup.py
  repositories/
    jobs.py
    history.py
    settings.py
  models.py
  errors.py
```

验收标准：

- 路由、业务逻辑、存储逻辑分离
- 测试能直接调用 service/repository
- `run.py` 启动方式保持兼容

#### T4-02 前端 JS 拆分

优先级：P2

问题：模板内嵌大量 JavaScript，长期维护和测试不方便。

建议：

- 将 JS 拆到 `static/js/app.js`
- 拆分 API client、单条转换、批量任务、音色选择、播放器
- 保留模板只负责 HTML 结构和静态资源引用

验收标准：

- `templates/index.html` 明显变薄
- 前端行为不变
- 后续可引入更完整的构建工具，但当前不强制

#### T4-03 结构化日志和诊断包

优先级：P1

建议：

- 日志记录请求 ID、job ID、voice、文本长度、耗时、错误码
- 本地日志滚动保存
- 诊断包包含日志、版本、系统信息、配置摘要、网络检测结果

验收标准：

- 用户反馈问题时能导出诊断包
- 日志不记录完整用户文本，避免隐私泄露

#### T4-04 API 契约文档

优先级：P2

建议：

- 文档化当前 API
- 说明请求参数、响应格式、错误码
- 后续任务开发先更新 API 契约再开发

验收标准：

- 新开发者能根据文档理解前后端交互
- API 变更有明确记录

#### T4-05 CI 和发布检查

优先级：P2

建议：

- 每次提交运行测试
- 打包前运行 lint/test
- release 前检查版本号、README、依赖、打包脚本

验收标准：

- 核心测试失败时不发布
- 打包步骤可重复

## 6. 推荐实施顺序

建议按下面顺序推进，能最快提升产品级稳定性：

## 9. 不建议优先投入的方向

以下方向不是没有价值，但不建议在底座稳定前优先做：

- 账号系统、云同步、订阅付费
- 大规模 UI 框架迁移
- Electron/Tauri 重写
- 多租户服务端部署
- 复杂音频编辑器
- AI 文案生成、脚本润色等偏离核心 TTS 的能力

原因是当前最重要的产品化收益来自稳定、可恢复、可试听、可管理的 TTS 工作流。底座稳住后，再考虑这些扩展会更轻松。

## 10. 成功衡量指标

建议后续用这些指标判断产品化是否有效：

- 转换成功率
- 平均失败重试次数
- 批量任务完成率
- 用户从输入文本到下载成功的耗时
- 长文本任务失败后可恢复比例
- 生成后试听率
- 重新生成率
- 历史记录复用率
- 常见错误的用户自助解决率
- 发布包安装成功率

## 11. 总结

当前项目已经具备清晰的核心价值，产品化不需要一开始就大改架构。最推荐的路径是：

1. 先补安全、任务、文件和测试底座。
2. 再补长文本分段、合并、取消、重试。
3. 然后补播放器、试听、历史、音色管理。
4. 最后提升桌面发布、设置、诊断和工程结构。

按这个顺序推进，可以在每个版本都交付可见价值，同时避免后续功能建立在脆弱基础上。
